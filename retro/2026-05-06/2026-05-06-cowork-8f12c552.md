# Concept Digest — cowork-8f12c552
_Session: "Fix obsidian graph view coloring" · 2026-05-06_

## Decisions

- **Graph color config is gitignored and ephemeral; the convention is version-controlled and canonical.** `.obsidian/graph.json` can be wiped by Obsidian. The cluster→color table in `wiki/.schema.md` is the durable artifact. The file is the projection; the schema is the source.
- **Projection mechanism created as a skill, not a script.** The `graph-colors` skill uses Read/Write tools with real file paths — no hardcoded session mount paths, no Python subprocess. Matches the relay-channel pattern.
- **Colors are explicit-trigger-only.** The skill runs only on user request. Never automatic. Reason: `graph.json` also holds forces, scale, and filter settings the user may be tuning; an automatic run would stomp live changes.

## Key Concepts

- **Source of truth vs. projection pattern.** A convention lives in a version-controlled document; a tool projects it onto an artifact on demand. The artifact is ephemeral; the convention is not. Recovery is always available by re-projecting. Applies wherever an OS or app writes config that can be lost.
- **Tag-query-based coloring (Obsidian).** Color groups are tag-query rules (`tag:#cluster/foundation → color X`), not per-file assignments. The convention auto-applies to future files carrying the right cluster tag — no post-hoc coloring needed.
- **Session-path vs. local-file path distinction.** Bash inside the Cowork sandbox sees files through a session mount (`/sessions/XYZ/mnt/…`). Read/Write tools operate on real local paths. Hardcoding session paths into scripts breaks across sessions — use file tools or vault-relative logic instead.
