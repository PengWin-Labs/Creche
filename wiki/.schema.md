# Wiki Schema Reference

Page formats, operation procedures, and conventions for this wiki. Loaded only when an operation triggers it — not part of session-start.

CLAUDE.md is session-law (Identity, Role Mapping, Session Start Protocol, pointers). This file is reference (page templates, operation steps, tagging, output formats).

---

## Page Formats

### Frontmatter (all wiki pages)

```yaml
---
title: Human-Readable Title
type: source | entity | concept | analysis | overview
tags: [tag1, tag2]
sources: [Source Slug 1, Source Slug 2]
created: YYYY-MM-DD
updated: YYYY-MM-DD
---
```

- `sources` tracks which raw sources contributed knowledge to this page
- `tags` should be consistent across pages — reuse existing tags before inventing new ones
- Always update `updated` when editing an existing page
- Every page carries exactly one `cluster/*` tag (see Tagging below)

### Source pages (`wiki/sources/`)

Filename: `YYYY-MM-DD Source Title.md` (date = ingest date)

```markdown
---
title: [Source Title]
type: source
tags: [...]
sources: [Source Title]
created: YYYY-MM-DD
updated: YYYY-MM-DD
---

## Summary
[2–4 paragraph synthesis of the source's key claims, arguments, or content.]

## Key Points
- [Bullet list of the most important specific facts, claims, or insights]

## Entities Mentioned
[[Entity 1]], [[Entity 2]], ...

## Concepts Discussed
[[Concept 1]], [[Concept 2]], ...

## Connections
[How this source relates to, confirms, or contradicts other wiki pages. Include explicit [[links]].]

## Raw Source
`raw/[filename]` (or note if raw was ingested-and-removed)
```

## Operations

### INGEST (legacy — `raw/` based)

Most ingestion now arrives via the relay-channel skill (`pebble-to-observer/`). This operation is the standalone path: a file dropped into `raw/` with the instruction "ingest [filename]".

**Steps:**
1. Read the source file from `raw/`.
2. **Launch Explore subagent** with: source path, `index.md`, hot-cache.md's recently-touched pointers. Prompt: "Survey this source against the wiki index. Without reading page bodies (use frontmatter and index entries only), return: (a) existing pages that need updating with one-line proposed change for each, (b) new pages that need creation with proposed slug + type, (c) entities/concepts mentioned that don't yet exist, (d) any pattern corrections that should propagate to personality.yaml or config.yaml. Cap at 250 words."
3. Brief discussion with Chris on takeaways from the subagent's report.
4. For each page in the subagent's list, read only that page and apply the targeted edit. Don't read pages not on the list.
5. Update `index.md` and append to `log.md`.
6. Update hot-cache.md (`last_file_activity`, **Recently Touched** pointers).

**After ingesting**, tell Chris: how many pages were created/updated, and what the most significant changes to the wiki were.

**Important:**
- One source may touch 5–20 wiki pages — that's normal
- When new source contradicts existing wiki content, flag it explicitly to Chris and note the contradiction on both pages
- Don't create entity or concept pages for things only tangentially mentioned

### QUERY

Triggered when Chris asks a question about the wiki's content.

**Steps:**
1. Check hot-cache.md's **Recently Touched** for relevant pointers first.
2. If the topic is recent, read those pages directly. If not, pull `index.md` to navigate.
3. Read the relevant pages (and follow links as needed).
4. Synthesize an answer grounded in wiki content, citing pages by name.
5. If the answer is non-trivial and worth keeping, offer to file it as an analysis page.

**Output format:** Match to the question. Simple questions get prose. Comparisons get tables. Multi-part questions may get headers. Always cite sources.

### LINT / CONSOLIDATE

Implemented by the `consolidate-dont-accumulate` skill (`.claude/skills/consolidate-dont-accumulate/SKILL.md`). Trigger phrases: "lint the wiki," "consolidate," "clean up the vault," "wiki cleanup." Two modes — `lint` (mechanical hygiene: dead links, orphans, index drift, frontmatter, fully-integrated sources) and `consolidate` (structural shape work: redundant pages, nested-cluster candidates, stubs, source retirement). Skill is the source of truth for procedure; this entry is a pointer.

Doctrine: working tree compounds shape; git holds the lineage. Deletion means deletion — no `_archive/` folders. See [[Git as Data Layer]].

### DRAFT

Triggered when Chris asks to "draft the README", "write a doc for X", or equivalent.

**Mode:** Document generation from wiki content. The wiki is the source of truth; the draft shapes it for a specific audience and purpose.

**Steps:**
1. Clarify: what is the document, who is the audience, what should it accomplish?
2. Read relevant wiki pages — overview, entities, concepts — to ground the draft.
3. Draft the document in the requested format (README, proposal, one-pager, etc.).
4. Lead with the most important thing for that audience. No preamble.
5. Offer to file the draft as an analysis page if it's worth keeping.

**Output defaults:**
- README → clear prose, "what it is / what it is not / why it works" structure
- Proposal → problem, mechanism, why this approach, what's needed to proceed
- One-pager → single thesis, three supporting points, one open question

## Index Conventions (`index.md`)

Organized by page type. Each entry is one line:

```
- [[Page Title]] — one-line description | `updated: YYYY-MM-DD` | sources: N
```

Update on every ingest, every new page created, and every lint pass.

## Log Conventions (`log.md`)

Append-only. Each entry:

```
## [YYYY-MM-DD] operation | Title or Description

[1–3 line note: what happened, what changed, any flags]
```

Operations: `ingest`, `query`, `lint`, `consolidate`, `draft`, `init`, `schema-update`, `skill-create`, `manual-edit`, `relay-integration`.

The `## [YYYY-MM-DD]` prefix is intentional — makes log entries greppable:
```bash
grep "^## \[" log.md | tail -10
```

`log.md` is the machine-readable record; redundant once git is tracking changes. Not the place for narration or shareable content — that's `dispatch.md`.

**Cleanup posture:** prune freely when bloated. Old entries don't need to be kept here — `git log -- log.md` has the lineage. Not skill-driven; just a discipline call when the file gets unwieldy.

## Dispatch Conventions (`dispatch.md`)

Human-readable build journal. Documentation written for an outside reader, in the founder's voice, as the work happens. Good documentation here is the first draft of content: posts, threads, README sections.

**The agent writes to it continuously during sessions** — not as a closing summary, but in the moment a decision worth explaining is made.

**What belongs:**
- Decisions and reasoning ("We collapsed 24 concept pages into 11 hubs because...")
- Tensions surfaced and how they were held
- Architectural insights as they land
- Corrections to prior thinking
- Context a future user, contributor, or reader would want

**What doesn't:**
- Operation tags and page counts (that's `log.md`)
- Frontmatter, links, or wiki formatting
- Summaries of what the agent did — narrate the *thinking*, not the *activity*

**Format:** Dated entries, plain prose, written for a smart person who wasn't in the room. No headers within entries. Length follows the weight of the thing being documented.

```
## YYYY-MM-DD

[Prose. Decision, reasoning, insight, or correction.]
```

Multiple entries on the same date are fine — append, don't merge. New date heading per session day.

**Cleanup posture:** none. Entries are historical record; the timestamp specificity is the value. If review tooling becomes useful (theme detection, content-harvest tracking, recurring-pattern surfacing), scope it as a separate skill rather than expanding `consolidate-dont-accumulate`. Cleanup and reflection are different modes of attention.

## Cross-Linking Rules

- Use `[[Page Title]]` Obsidian-style wikilinks throughout all wiki pages
- Every entity and concept mentioned significantly should be linked
- Source pages link to every entity and concept they introduce or discuss
- Entity and concept pages link back to every source that mentions them
- Analysis pages link to every page they synthesize from
- `overview.md` links to the most important entities and concepts only — don't over-link

## Tagging Conventions

Two layers of tags. Cluster tags drive graph coloring; descriptive tags are free-form.

### Cluster tags (required, exactly one per page)

| Cluster tag | Hex | Pages |
|---|---|---|
| `cluster/foundation` | `#1e3a8a` deep navy | Foundational Principle, Connections vs Solving, Founder as First User |
| `cluster/doctrine` | `#2563eb` royal blue | Doctrine Practice Expression, Guardrails |
| `cluster/architecture` | `#0e7490` teal | Three-Tier CRM Model, Three Layers of Data, Read-Path Write-Path Split, Git as Data Layer, Package JSON for a Person |
| `cluster/process` | `#65a30d` olive | Alignment |
| `cluster/privacy` | `#4338ca` indigo | Privacy as Structural Property |
| `cluster/agents` | `#d97706` amber | Crèche, Pebble, Observer, Cache |
| `cluster/social` | `#059669` emerald | Rookery, Journal |
| `cluster/business` | `#166534` forest green | Pengwin Labs, Forage |
| `cluster/sources` | `#9ca3af` neutral grey | Crèche Prior Art, Crèche Vault |

A page belongs to exactly one cluster — the load-bearing one. Cross-cluster relationships travel through wikilinks, not multiple cluster tags.

### Descriptive tags (free-form, optional)

Anything that helps queries / dataview without driving color: `framework`, `philosophy`, `architecture`, `privacy`, `doctrine`, `accessibility`, `personal-crm`, `agent`, `technology`, `business`, `legal`. Reuse rather than invent.

### Status tags (when relevant)

`stub`, `well-developed`, `contradicted`, `superseded` — apply only when the page is genuinely in one of these states.

### Future: nested cluster tags (v2)

Hierarchical tags like `cluster/architecture/data-shape` would let nested pages render as lighter shades of their parent cluster color. Not implemented in v1. Revisit when the wiki has enough subgroupings to justify.

## Output Formats

Default output for queries is markdown prose with `[[links]]`. Other formats available on request:

- **Table** — for comparisons (`| Col | Col |`)
- **Marp slide deck** — for presentations (add `marp: true` to frontmatter)
- **Dataview query** — for dynamic Obsidian tables based on frontmatter
- **Plain list** — for quick inventories
