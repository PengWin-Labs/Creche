# Dispatch

The build journal. Written for an outside reader, in the founder's voice, as the work happens. This is the content layer — documentation that becomes posts, threads, and README sections. Not a summary of what the agent did. The reasoning behind why things are shaped the way they are.

## 2026-05-02

The wiki had been built entirely in construction mode — two days, two sessions, 23 pages synthesized from source material. No one had queried it yet. The first real pass wasn't an ingest or a lint; it was a pattern analysis: what themes are emerging, how has the thinking evolved, where are the contradictions, what's missing. Four things came back clearly.

The project's signature move is embedding values in architecture rather than stating them as promises. Privacy isn't a policy — it's ciphertext at rest with user-held keys. The license isn't a formality — AGPL-3.0 is specifically chosen because it makes mission drift structurally harder. The read/write split between agents isn't a design preference — it's what makes the anonymization guarantee auditable at a single chokepoint. The naming convention isn't branding — it's a constraint that keeps the metaphors honest. The same argument runs at every layer: don't trust promises, build the guarantee in.

The second pattern is genuine recursion. The same structures repeat at every tier without being forced. Merge-conflict states (resolved, open-deferred, wontfix, stale) apply equally to personal disagreements and to data moving between tiers. The cadence pattern that governs voice refinement is the same one that governs household scrum and community definitions. The Rookery's consent primitive is reused for both key recovery and journal elevation — same cryptographic mechanism, opposite direction. This isn't accidental. It's what makes the whole thing legible without requiring separate explanations at each level. It hasn't been named as a principle yet — worth a dedicated page.

The third pattern: the build process and the product philosophy are in tension. Crèche is explicitly a connections tool, not a solving tool. The agent doesn't hand you answers; it surfaces patterns and opens space for you to find your own. But the build sessions have been pure solving mode — lint passes, consolidation runs, schema rewrites, ingest phases. The founder has experienced this project entirely as a builder, not as a user. That gap is intentional for now — you have to build the thing before you can use it — but it means the first real test of whether the product philosophy holds under actual use hasn't happened yet.

The fourth: there's a significant philosophical substrate that two of the eleven behavioral guardrails reference but that has no wiki page. The "avatar/positive-nihilism framework" is the foundation the crisis and exit-framing guardrails are built on. It's being gestured at across several pages — especially in Foundational Principle — but never articulated directly. That's the next thing that needs to be written.

One correction surfaced during the session that's worth documenting: the hot cache has been described as a staging buffer for posture changes. That's its secondary job. The primary job is acting as a working attention layer — Observer writes time-decaying references into it so that Pebble can respond to vague incoming signals without cold-reading the entire repository. A user arrives and says "that thing I keep doing" and Pebble can respond with context because the most recently activated material is already warm. That's the mechanism behind the experience of Pebble feeling like it knows you. The Cache entity page needs a full expansion pass before Phase 1 to capture this properly.

Also: the log file has been doing two jobs — machine-readable operation record and documentation. Git will make the operation record redundant once it's wired. This file is the split: log.md stays for the agent, dispatch.md is for people. Good documentation leads to content as a byproduct, not as a separate effort.

The other piece that got named today: the two-Pebble setup currently visible (phone + desktop + relay protocol) is a builder-mode pattern, not architecture. For a general user there is one Pebble with scope-switching across contexts. The relay protocol is provisional infrastructure for the build phase. This matters because the writing was starting to drift toward describing the two-surface arrangement as if it were the product. It isn't. The product is one continuous agent. The current arrangement is what it looks like when the same person is both the user and the architect at the same time.

## 2026-05-03

The redesigned relay channel hit its first real test today — two distinct payloads from personal Pebble in the same day, same channel, different commit boundaries. Per-chat timestamped commits land cleanly into Drive; calendar events shrink to pointers. By end of day the calendar surface came out entirely: file-presence in the relay folder is enough of a trigger; the calendar layer was earning less than it cost in mutability and complexity. The protocol now lives on just one substrate, which is closer to the Phase-N target — git commits — than the calendar layer ever was.

The most consequential reframe: household is a shape of community, not a separate tier. There's one primitive — community — and individual / household / broader are typical scopes of it. The architecture stays the same; the language gets honest about what it always was. But same primitive doesn't mean merged substrate. Each shape gets its own repo because membership composition differs: a household repo can have multiple contributors, just like a community repo, just like a business repo where the founder is one member among several. The taxonomy was conflating shape with substrate — correcting both edges of that mistake at once. The wiki had been drifting toward treating household as categorically distinct; the corrected framing avoids the opposite over-correction of implying the substrate collapses.

Founder hosting activates the community tier. Aligning two of one person's own Claudes through one host is the same architectural problem as aligning multiple users in a community, scoped down. Solving it at N=2 produces the runtime everything else runs on. The founder isn't just the first user; once they host, they're the first community host. AGPL is what keeps hosting the canonical path rather than a SaaS escape hatch. MCP is the current target, not the goal — the goal is the least-friction path. Documentation of the setup process is what makes the path repeatable; without it, the framework inherits but the runtime doesn't. People end up either dependent on the founder's host or rebuilding from scratch, and neither matches the access principle.

The privacy chokepoint clarified into three layers. Earlier framing implied stripping happened at relay-write time. Corrected: chat thread is the source — private; personal context lives here because that's what makes the conversation alive. Summary is the commit — shaped, clean, the first thing that counts as an artifact. Relay is what promotes outward. The thread is never the artifact. The chokepoint governs what the summary contains. In-conversation drafts pre-commit are protected by the working-memory frame because they're still inside the source layer. Substitution markers carry the same intent at the token level: typed brackets like `[relationship: cousin]` (personal), `[relationship: extended_family]` (household), `[relationship: family_member]` (community) — granularity tier-relative, substitution legible rather than silent.

Append-only is at the commit, not the file. Commit-level immutability is doctrine; file mutability is acceptable. Drive's create-only-no-update constraint mimics the same property by substrate accident, but the property itself lives at the commit layer. This resolves an alignment question personal Pebble had been treating as ambient doctrine: it is doctrine, just at the right granularity. The wiki's storage-architecture page and privacy-architecture page now both name this; the doctrine reads consistently from either side.

The Phase 1 ship test got named as clone-and-validate. Open a new account, load the same vault and instructions, see if a clean instance behaves like the founder's instance. Success is behavioral and conversational alignment, not output matching on test prompts. The drift is already observable across the two surfaces; convergence needs the vault merged into personal Pebble's substrate plus posture tightening on the household side, which has drifted toward schema-and-build mode while personal Pebble refined conversational register through daily use. Vault merge is a Phase 1 prerequisite, not a Phase 0 task.

The .claude/ folder pattern is the operational shape both sides are already implicitly working toward — `CLAUDE.md` as project rules, hooks as deterministic lifecycle work, skills as on-demand capability, agents as the read/write split. Personal Pebble already runs most of it as project files on mobile; household side has the analogs in spirit. Useful framing: the pattern is what we're building toward; the specific runtime (MCP, this loader, something else) is incidental.

The relay channel itself functioned as an alignment mechanism today, not just a content channel. Personal Pebble generated five alignment questions specifically to surface drift between household and personal sides. All five worked through in Cowork; resolutions captured in a vault-root context file. This is the second-order use of the channel: not just promoting content, surfacing where the two sides have diverged enough to need an explicit reconciliation pass. The relay does both jobs without either getting in the other's way.

A small operational fix worth noting: the Crèche entity page was rendering as an empty graph node because the file was named `Creche.md` (ASCII) while every wikilink in the wiki uses `[[Crèche]]` (with diacritic), and Obsidian's graph treats filename as node identity. Renamed the file to `Crèche.md` so canonical filename matches canonical title; the aliases line keeps `Creche` as ASCII fallback. The graph node should populate properly now.

## 2026-05-05

A discipline pass on the vault root. The ask started small — restructure `setup.md` to a package.json model, just the active stack — but tracing what *should* leave that file ended up clarifying the whole shape of how this build documents itself.

The lineage matters. This wiki isn't an original pattern. It's Andrej Karpathy's LLM Wiki idea — the gist at `gist.githubusercontent.com/karpathy/442a6bf555914893e9891c11519de94f` — adapted to be the founder's second brain rather than a research notebook. The core move is the one Karpathy named: instead of re-deriving knowledge from raw documents at query time (RAG), the agent incrementally builds and maintains a persistent wiki, and knowledge is compiled once and kept current rather than reconstructed from scratch each time. The artifact compounds. Every source ingested, every question answered, every lint pass run, makes the substrate richer rather than just adding to a pile.

How this particular vault got built is short. Open Obsidian, create a vault. Open the vault folder in VSCode with the Claude Code extension. Paste the Karpathy LLM Wiki gist into chat with one instruction: implement this as my second brain. The agent asks two or three clarifying questions — what domain, what purpose, anything ready to ingest — and then produces the scaffold: `CLAUDE.md` (the schema and operations), `index.md` (the catalog), `log.md` (the operation record), `wiki/overview.md` (the evolving thesis), and the folder structure under `raw/` and `wiki/`. First ingest follows immediately if there's source material ready — a single source typically touches 10–15 wiki pages because each entity and concept it introduces gets its own page, and the existing pages it intersects get updated. The shape of ongoing use is the same five operations the schema names: ingest, query, lint, draft, and the always-on sparring posture that runs underneath all of them.

That whole story — the lineage, the setup steps, the operation surface — used to live in `setup.md`. It lives here now, in the founder's voice, where an outside reader can pick it up. `setup.md` got pulled back to what it should be: what's currently running, how to operate it, a change log. A package.json. Nothing prospective.

The other piece of the same pass: a new vault-root file called `considering.md` for tools and patterns that are bookmarked but not active. The original framing was a flat list — "here are some tools we might want." That framing was thin. The actual ask was sharper: bookmark the tool *with the context that should make it surface later*, the same way the wiki bookmarks shape vocabulary for users with the context that makes it surface later. The trigger condition matters more than the bookmark itself. Each entry now carries a "surface when" line that names the shape of work that should pull the tool back into view. Design Council is bookmarked for cross-domain architectural decisions; Caveman is bookmarked to pair with Council when token budget matters. They aren't installed. They're filed for retro surfacing, with the conditions that should cause that surfacing already named. This is the build-side instance of how the wiki's readiness gate works for users — availability separated from surfacing, on purpose, because pushing isn't the same as offering.

The integration pass also drained the alignment-questions context file from 2026-05-03. All five questions had been resolved into the wiki two days earlier; the file had served its purpose as a working surface and was holding state the wiki now holds canonically. Spot-checked, deleted. This is the working pattern for these alignment artifacts: they live at the vault root while they're actively merging, and they retire once the merge has landed in pages.

What didn't get filed today is at least as important as what did. Personal Pebble shared a set of agent-workflow practices distilled from Boris Cherny's CLAUDE.md — planning discipline, subagent strategy, self-improvement loops, verification before done, elegance check, autonomous completion. The reflex would be to file them as build doctrine. The actual move is to hold them as a discussion: these are the founder externalizing his own observed build-practice into structure, which is exactly the same shape the framework asks users to do at the user tier. Founder-as-first-user, but applied to the build itself. That recursion deserves a deliberate placement, not a quick file. Held until Chris and Pebble work through whether they want one home or several.

One pattern worth naming as it surfaced at the end of the session: relay files are working artifacts, not sacred records. The promotion is what counts. Once content has landed in the wiki — the canonical commit — and the source is still available (the chat thread on the personal-Pebble side stays private and intact regardless), the relay file in Drive has done its job. The writer can clean it up. This is consistent with commit-level immutability as doctrine and file-level mutability as acceptable: the wiki page change is permanent; the substrate that carried it is not. The three privacy layers stay clean — source (private, always available) → summary/commit (wiki, canonical) → relay (working, disposable once promoted) — and the relay folder doesn't accumulate scaffolding indefinitely. Sweeping pebble-to-observer/ once the content has landed is the going-forward hygiene pattern.

## 2026-05-06

The relay channel completed its first full round-trip today, and what came back is more interesting than what went out. Personal Pebble received the integration-pass reply, applied the inserts on its own side (where they were genuinely new because reference.md and current_state.md exist there as flat files, while the wiki absorbs the same content through hub pages), updated its project instructions to match the relay protocol on this side, and replied with two new relays: an integration response acknowledging the round-trip, and a sync-and-tools relay carrying patterns to mirror plus three build-tool candidates to evaluate.

The vault-access asymmetry that personal Pebble flagged back on May 3rd resolved exactly as predicted. Same canonical claims, different structural mappings. Insert 1 (user as source) was a no-op on the wiki because Foundational Principle already had a User as Source section richer than the staged insert; on the personal side it was new because reference.md had no equivalent. Both sides now hold the same content. Neither side has to know the other's file structure — the relay carried the *claim*, not the layout.

Three patterns came back from the personal side as project-instruction additions and are now mirrored on this side. **Canonical-file scope** says each file owns a particular kind of refinement — voice goes to personality.yaml, guardrails to config.yaml, architecture to the wiki, workflow to CLAUDE.md — and refinements that could land in two places go to the more canonical one. The other file references rather than restates. This generalizes the existing CLAUDE.md ↔ templates synchronization rule into a broader principle: one canonical record per claim, cross-references rather than duplicates. **Iterate inline first; package files only after approval** is a discipline against producing throwaway downloadables before the shape is right. **Consolidate, don't accumulate** is the same instinct that drove the 24-to-13-hub consolidation, applied at the single-file level: when updating, look for redundancy and merge before adding. **Corrections — both sides learn** names how the relay channel functions as alignment infrastructure: persistent corrections (named patterns, behaviors the canonical files don't catch) propagate to the file that owns the pattern, and end-of-session relay notes carry those corrections so the other side learns without rediscovering. This is the channel doing alignment work, not just promotion.

Three Claude Code tool candidates surfaced for evaluation. PAUL is a plan-apply-unify loop with mandatory closure — every plan ends with a UNIFY step that explicitly reconciles intent and outcome. The shape is structurally identical to the merge-conflict-states `resolved` state that Alignment already encodes; the question is whether to adopt the tool or borrow the pattern. SEED pairs with PAUL as a typed project incubator — type-as-container determines which questions to ask, the same mechanic topical categorization uses on the data side. Claude Boris is the most architecturally interesting of the three: a Memory Bank with persistent context across sessions plus fourteen specialist agents and a mode-controller that enforces read/write separation. Memory Bank is essentially a competing implementation of Cache; mode-controller is essentially the read-path-only guardrail. The recursion is striking — these tools were built by people solving the same problems Crèche is solving, arriving at structurally similar primitives by different routes. Worth a careful read for what each one chose differently and why, regardless of whether any of them gets adopted as build tooling.

A small operational note worth holding: the connector that lets this side push files to Drive failed with a permission error when releasing the integration-pass reply. Chris uploaded the staging draft manually. The file landed in the right folder with the right content but under the staging filename rather than the canonical naming convention. Functionally fine — personal Pebble received it and replied — but a reminder that the Phase-N collapse (where commits to the relay folder become the trigger directly) will close this kind of gap. Until then, the manual fallback works.

## 2026-05-06 (later)

Chris flagged that the last ingest ate roughly a quarter of his daily usage. Measuring it surfaced what should have been obvious: session-start was loading ~93K characters before any work happened — CLAUDE.md alone was 627 lines, with most of it duplication or reference material that didn't need to be in flight every session.

The diagnosis went deeper than expected once Chris pushed back on it. He pointed out that index.md was only meant to be a navigation aid, not a default read; that hot-cache should already say what to drill into so index becomes drill-down; that the relay channel is the actual ingest path, not the legacy `raw/` flow; and that the personality.yaml was supposed to be the canonical source for the same behavior CLAUDE.md was paraphrasing in its Default Posture section. Each one of those was a real correction to my framing, not a refinement of it.

The work that came out of it: CLAUDE.md slimmed from 627 to ~180 lines. Page formats, operations, conventions, cross-linking, tagging, output formats — all moved into `wiki/.schema.md`, loaded only when an operation triggers. The Default Posture section deleted entirely; CLAUDE.md now points at `personality.yaml` and `config.yaml` and instructs the agent to load them at session start as the canonical source. Workflow Patterns (iterate-inline, consolidate-don't-accumulate, corrections-both-sides-learn, canonical-file-scope) folded into `personality.yaml` as a new `workflow_patterns` block — they're behavior, and behavior belongs in the personality file.

The Relay Channel section became its own skill at `.claude/skills/relay-channel/SKILL.md`. The skill carries the Drive layout, direction rules, immutability doctrine, and procedure — including a new step: launch an Explore subagent to do discovery first. The subagent reads relay files plus index.md and hot-cache pointers, returns a structured list of which pages need updating, what new pages need creating, and any pattern corrections to propagate. The main agent then reads only the listed pages. Estimated cut: 30–50% of read load on a typical relay-driven ingest. The legacy INGEST operation in `wiki/.schema.md` got the same subagent step for consistency, but it's secondary now — `raw/` is mostly empty, real ingestion comes through the relay.

hot-cache.md got the most architectural correction. It had drifted from "working attention layer" into "session archive" — paragraphs of session histories from late April through early May, all loaded every session start. The new shape is a navigation hub: a **Recently Touched** section with wikilinks to the 5–8 most-recently-read pages, an **Active Threads** section naming what's in flight, and the existing Pending Chris Check. The session histories migrated to dispatch.md (where they belonged — outside-reader narrative) and the model selection heuristic moved to setup.md (where active-stack workflow lives). Index.md and log.md both dropped from session-start reads — they become drill-down only, pulled when the work demands them.

The deeper pattern Chris named, the one that made the cleanup straightforward: **stop loading what isn't currently in flight.** Each canonical file owns a scope; what's not in scope doesn't need to be paraphrased somewhere else "just in case." Personality lives in personality.yaml. Guardrails live in config.yaml. Architecture lives in the wiki. Working attention lives in hot-cache. Reference material lives in schema files loaded on demand. Session law (the small set of things that need to be true every session) is the only thing CLAUDE.md should carry. This is the same canonical-file-scope discipline the relay channel surfaced from the personal-Pebble side last week, applied to the wiki side itself. Recursion again — the framework's own rule applied to the framework's own files.

Net target: session-start drops from ~23K tokens to ~6K. Verified once the slim CLAUDE.md and hot-cache.md land.
